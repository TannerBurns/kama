# Security Policy

## Supported versions

Kama has not published a supported release yet. Until the first release, security
fixes are made on a best-effort basis on the `main` branch. A supported-version table
will be added when release support begins.

## Reporting a vulnerability

Do not disclose a suspected vulnerability in a public issue, discussion, or pull
request. Use the repository's
[private vulnerability reporting form](https://github.com/TannerBurns/kama/security/advisories/new).
If that form is unavailable, ask the repository owner for a private reporting channel
without including vulnerability details in the request.

Include, when possible:

- the affected version, commit, component, and configuration;
- reproduction steps or a minimal proof of concept;
- the impact and conditions required for exploitation; and
- any suggested mitigation or remediation.

Maintainers will acknowledge receipt, establish a private communication channel,
assess severity and affected versions, and coordinate a fix and disclosure. Please
allow a reasonable remediation period before public disclosure. The project will
credit reporters unless they request anonymity.

Security boundaries and threat assumptions are documented in
[Operations, Security, and Observability](docs/project-plan/07-operations-security-and-observability.md).
