## CPU serving and failure evidence

Run the production CPU runtime suite:

```sh
make test-e2e-serving-cpu K8S_MINOR=1.36
make verify-e2e-serving-cpu-evidence K8S_MINOR=1.36
```

It must use the pinned loadable SmolLM2-360M-Instruct GGUF and production CPU image,
then retain redacted evidence under `dist/e2e/serving-cpu`. One run must prove:

- the artifact UID/digest/location gates workload creation and the mounted path is
  read-only;
- a bounded Pod using Kama's static serving client reaches the model through its
  ClusterIP Service DNS, observes nonempty generated assistant content in a valid
  OpenAI-compatible SSE chunk, and then observes the terminal `[DONE]` event;
- the CPU runtime image evidence contains a BuildKit-produced immutable manifest
  digest and OCI revision/source labels matching the checkout; a functional run with
  no recoverable manifest digest records `qualifying=false`, and hosted CPU jobs
  reject that result. The disposable Kind cluster loads those content-addressed
  Buildx images under run-local tags with `imagePullPolicy: Never`; the recorded
  manifest digests and matching OCI labels, not those isolated transport tags, are
  the immutable qualification identities;
- CPU mode reports `Serving=True` and `Degraded=True` with
  `CPUOnlyRequested`;
- delayed loading creates no ready EndpointSlice address until the child is ready;
- an invalid artifact or terminal load failure leaves the supervisor running,
  unready, with container restart count zero and no ready endpoint;
- update/deletion observes the first SSE event, proves the request is still active
  and incomplete, then observes Pod and EndpointSlice readiness removal before the
  stream completes within the declared deadline; and
- a changed artifact identity drains the loaded workload, remains stable without a
  replacement Pod or ready endpoint, then restores the original artifact behind the
  same Service and serves a second generated-content SSE request from a new Pod.

## Protected NVIDIA acceptance gate

Run only trusted `main` code on the protected runner/cluster:

```sh
make test-e2e-serving-nvidia
```

The runner labels are `[self-hosted, Linux, X64, kama-nvidia]`. Prefer an ephemeral
runner; otherwise configure its runner group and cluster credentials for this
repository only. Restrict the protected `e2e-nvidia` environment to the `main`
deployment branch with required reviewers. The environment provides a dedicated
kubeconfig for a supported Kubernetes 1.34, 1.35, or 1.36 cluster without exposing
it to pull requests. The workflow writes it to a
run-specific mode-`0600` file and removes it in an unconditional cleanup step. A
hosted validation job rejects manual dispatches targeting any non-`main` ref, so a
skipped GPU job cannot produce a false-green qualification. The protected job checks
out the exact
`E2E_NVIDIA_EXPECTED_COMMIT` with full history and requires that commit to be an
ancestor of the current `origin/main` before it receives the kubeconfig or uses the
GPU cluster. Make the stable `Protected NVIDIA acceptance gate` check required for
M2 promotion.
The suite must retain `dist/e2e/serving-nvidia` and prove:

- the generated Pod requests and receives exactly one full `nvidia.com/gpu`;
- the CUDA runtime observes exactly one visible GPU, reports a nonempty device
  identity, and reports `offloadedLayers == totalLayers > 0`;
- a bounded in-cluster Pod using the immutable Kama serving-client fixture reaches the
  verified model through its ClusterIP Service DNS, observes nonempty generated
  assistant content in a valid OpenAI-compatible SSE chunk, and observes the terminal
  `[DONE]` event;
- `RuntimeReady` and `Serving` describe the current artifact/runtime fingerprint,
  status names and UIDs match the exact artifact, Deployment, and Service objects,
  desired/observed/loaded fingerprints agree, desired and observed CUDA image
  identities agree with the selected manifest, effective context/concurrency are
  `2048`/`1`, desired/ready replicas are `1`/`1`, and the Pod has no unexpected
  restarts;
- the suite queries the allocated container for its actual GPU and driver, reads the
  installed CUDA version, validates the installed CUDA 12.4 runtime package, and
  verifies that `llama-server` links to both CUDA runtime and BLAS libraries instead
  of trusting supplied labels. The evidence records those facts, Kubernetes,
  llama.cpp commit, manager/runtime/model digests, sanitized state, workload
  security/resources/image identity, Pod UID/node/readiness/restart count,
  EndpointSlice readiness, and request metadata; and
- every production image and the serving-client fixture image have OCI source and
  revision labels matching the trusted checkout, both runtime images match the
  llama.cpp pin, and the CUDA image matches the CUDA pin. Each reference must use its
  canonical `ghcr.io/tannerburns/kama-*` repository at an immutable digest. Cosign
  must verify a keyless release signature and SPDX JSON attestation for every image,
  bound to Kama's trusted release workflow and the expected commit. Missing
  inspection, signature, attestation, CUDA, or linkage evidence retains the
  functional result with `qualifying=false` and fails qualification.

## Completion rule

Repository deliverable checkboxes may be checked once that deliverable is implemented
and its automated repository gates pass; an unchecked runtime-image deliverable remains
pending until the production images are built and qualified. M2 remains `In Progress`
until all repository, CPU/failure, and real-NVIDIA evidence tables are complete. Then
update the delivery roadmap in an evidence-only follow-up:

1. check any remaining deliverables and all three definition-of-done entries;
2. set `Status: Complete (YYYY-MM-DD)`;
3. link the immutable implementation commit and CI, Kind, CPU, and NVIDIA runs; and
4. record the published CPU/CUDA image and model digests.

M1 fixtures are prerequisites, not M2 evidence. A successful fake llama-server test,
CUDA image build, or protected workflow definition does not qualify the NVIDIA gate.
