implementation, fixture-backed cluster validation, the pinned public Hugging Face E2E

## Repository implementation and local verification

Run the non-container repository checks from a clean checkout:

```sh
make verify-generated
make test
make test-envtest
make helm-validate
```

The repository contains the M1 APIs, webhooks, controllers, importer, generated
manifests, chart, examples, fixtures, and runbooks. Its automated tests cover the
implementation contract, including:

- path traversal and symlink defenses, GGUF and manifest goldens;
- HTTP Range and ETag resume, immutable revision pinning, publication recovery, and
  bounded result parsing;
- credential redaction, deterministic Jobs, Lease recovery, and controller status
  transitions;
- identity-scoped artifact cleanup and guarded managed-claim deletion; and
- bounded-label Events and metrics.

These checks establish repository implementation; they do not establish the
self-contained artifact-plane E2E result or a named production CSI qualification
result. The clean run is linked below.

## Fixture-backed Kind matrix

Run each supported Kubernetes minor from a clean checkout:

```sh
for minor in 1.34 1.35 1.36; do make test-kind K8S_MINOR="$minor"; done
```

The Kind harness is implemented as a two-node cluster using project-owned GGUF and
Hub fixtures, static RWO local volumes, and the release chart. It is designed to
verify:

- CRD installation and fail-closed defaulting/validation webhooks;
- a managed cache probe and retained managed claim;
- public and token-protected Hub imports with one-download evidence across manager
  restart;
- manual PVC `Copy` and `Direct`, including `SingleNode` and PV node-affinity status;
- checksum, unauthorized-source, and missing-shard conditions remaining non-ready;
- expected manager/importer image references and executable version probes, the
  rendered manager and representative Hub-import Job restricted security defaults,
  representative emitted artifact/cache metrics with expected bounded labels, chart
  RBAC for guarded PVC deletion, semantic Direct/cache admission rejection,
  server-side validation of every example, KEDA activation, and chart cleanup without
  PVC deletion. The packaging gate independently compares binary/image versions with
  the repository release version.

The presence of this harness is not run evidence. Its static local volumes also do
not qualify a production CSI driver or replace the artifact-plane E2E acceptance
suite. Record any unexecuted or uncovered item as pending rather than treating this
list as proof.

## Self-contained artifact-plane E2E and private-source follow-up

The reusable E2E workflow creates its Kubernetes clusters and functional storage
services inside CI runners. Outbound access to Hugging Face is its only input for the
ordinary scheduled regression. A protected, read-only token and immutable private
artifact metadata are additional inputs for the non-blocking private-source follow-up.
That private scenario runs only for a trusted manual dispatch that explicitly enables
it; a run where the scenario is skipped is not qualifying private-source evidence.

The implementation is the artifact-plane profile in the
[end-to-end workflow](../../.github/workflows/e2e.yml), with suite conventions in
[the E2E test README](../../test/e2e/README.md). It always exercises the public
[`HuggingFaceTB/SmolLM2-360M-Instruct-GGUF`](https://huggingface.co/HuggingFaceTB/SmolLM2-360M-Instruct-GGUF)
fixture. Configure the `e2e-huggingface` GitHub environment with a fine-grained
read-only `HF_TOKEN` Secret and the following non-secret environment variables for a
qualifying private lane:

| Variable | Value |
|---|---|
| `E2E_PRIVATE_HF_REPOSITORY` | Project-controlled private `owner/repository` |
| `E2E_PRIVATE_HF_REVISION` | Full immutable 40-character commit |
| `E2E_PRIVATE_HF_FILE` | One selected GGUF path |
| `E2E_PRIVATE_HF_SHA256` | Expected lowercase file SHA-256 |
| `E2E_PRIVATE_HF_SIZE` | Expected file size in bytes |

A trusted `main` push, scheduled run, or ordinary manual dispatch may pass its
public-only job while recording `qualifying=false`; it is useful external-source
regression evidence but does not qualify the private-source follow-up. A manual
dispatch with `require_private_hf=true` requires the protected configuration and fails
closed when it is absent or invalid. The Hugging Face job runs only from `main`, so
protected credentials are never provided to arbitrary branch code.

A future qualifying private-source follow-up should rerun the complete contract and
prove all of the following:

- import an immutable public GGUF revision from Hugging Face and an immutable
  project-controlled private Hugging Face revision, checking the declared size and
  SHA-256 digest for each;
- interrupt an importer during transfer and prove resumable staging eventually
  publishes the artifact; after it is ready, restart the manager and force another
  reconciliation, proving readiness is preserved and no artifact bytes transfer;
- mount functional RWX storage from Pods scheduled to both nodes concurrently and
  verify identical read, checksum, and `mmap` results;
- validate a functional adopted RWO artifact and prove its reported PV node affinity
  agrees with the consuming Pod's placement;
- exercise corrupt, incomplete, unauthorized, and genuinely exhausted functional
  storage paths, proving that each remains non-ready and cannot alter an existing
  ready publication; and
- retain redacted statuses, Events, bounded metrics, restart timestamps, storage and
  Kubernetes versions, and scenario logs as workflow artifacts.

Hugging Face repository download counters are aggregate and are not a deterministic
per-run assertion. The workflow must instead retain runner-scoped source-request
evidence and importer transfer metrics, then make the source unavailable before the
post-restart reconciliation so a cache hit cannot be mistaken for another download.
Tokens, authorization headers, Secret values, and authorization response bodies must
never be captured.

Passing a test CSI driver or in-runner storage service establishes the artifact-plane
storage semantics required by M1 only. It does not create a support claim for that
driver, backend, mount configuration, availability model, durability, or throughput.

## Owner closure and non-blocking follow-up

The project owner closed M1 on 2026-07-20 using the immutable repository, public
Hugging Face, Kind, and storage/failure evidence above. This decision accepts the
absence of a project-controlled private-source run without representing it as a pass
or weakening the workflow's `qualifying` signal.

A future qualifying private-source run should record:

- the exact repository commit and immutable workflow/run/job URLs;
- the public and private model repository, revision, path, size, and SHA-256 digest;
- the Kubernetes, test storage driver/service, and container image versions; and
- downloadable redacted artifacts for every scenario above.

Until then, the follow-up remains visibly outstanding but does not reopen or block
the completed M1 milestone.

## Named production storage compatibility gate

Qualification of a real production CSI driver and backend is deliberately separate
from M1. A protected workflow or repeatable operator-run procedure must exercise the
same storage floor against the actual driver/backend and enter measured results in
[storage qualification](../storage-qualification.md). Those results are required
before documentation or a release claims support for that named configuration, but a
missing named result does not reopen or block an otherwise completed M1 milestone.
