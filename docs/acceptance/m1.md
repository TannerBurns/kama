## Repository implementation and local verification

Run the non-container repository checks from a clean checkout:

```sh
make verify-generated
make test
make test-envtest
make helm-validate
```

The repository contains the M1 APIs, webhooks, controllers, importer, generated
manifests, chart, examples, fixtures, and runbooks. Its automated tests cover the
implementation contract, including:

- path traversal and symlink defenses, GGUF and manifest goldens;
- HTTP Range and ETag resume, immutable revision pinning, publication recovery, and
  bounded result parsing;
- credential redaction, deterministic Jobs, Lease recovery, and controller status
  transitions;
- identity-scoped artifact cleanup and guarded managed-claim deletion; and
- bounded-label Events and metrics.

These checks establish repository implementation; they do not establish a Kind
result, a live Hugging Face result, or a production CSI qualification result. Record
links only after a clean run completes.

## Fixture-backed Kind matrix

Run each supported Kubernetes minor from a clean checkout:

```sh
for minor in 1.34 1.35 1.36; do make test-kind K8S_MINOR="$minor"; done
```

The Kind harness is implemented as a two-node cluster using project-owned GGUF and
Hub fixtures, static RWO local volumes, and the release chart. It is designed to
verify:

- CRD installation and fail-closed defaulting/validation webhooks;
- a managed cache probe and retained managed claim;
- public and token-protected Hub imports with one-download evidence across manager
  restart;
- manual PVC `Copy` and `Direct`, including `SingleNode` and PV node-affinity status;
- checksum, unauthorized-source, and missing-shard conditions remaining non-ready;
- expected manager/importer image references and executable version probes, the
  rendered manager and representative Hub-import Job restricted security defaults,
  representative emitted artifact/cache metrics with expected bounded labels, chart
  RBAC for guarded PVC deletion, semantic Direct/cache admission rejection,
  server-side validation of every example, KEDA activation, and chart cleanup without
  PVC deletion. The packaging gate independently compares binary/image versions with
  the repository release version.

The presence of this harness is not run evidence. Its static local volumes also do
not qualify a production CSI driver or replace the strict live storage gate. Record
any unexecuted or uncovered item as pending rather than treating this list as proof.

## Strict live gate

The following needs real credentials and CSI infrastructure and is intentionally not
substituted by the local fixture suite:

- a pinned public and gated/private Hugging Face repository, including source-side
  download-count evidence across manager and importer-Job interruption;
- a real RWX CSI cache read concurrently from Pods on two nodes;
- a real adopted RWO artifact whose reported PV affinity matches scheduling;
- genuinely exhausted storage plus corrupt and incomplete content, proving no ready
  publication is damaged;
- measured probe/import/load results entered in
  [storage qualification](../storage-qualification.md).

For each scenario retain the input manifests with tokens removed, resource status,
Events, relevant bounded metrics, Kubernetes/CSI versions, restart timestamps,
source download counts, and CI or operator log links. Do not capture Secret data or
authorization response bodies.

Only after every repository/Kind row and strict scenario has immutable evidence may
the delivery roadmap change M1 to `Complete`.
